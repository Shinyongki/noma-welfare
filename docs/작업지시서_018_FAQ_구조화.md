## 1. 목표

고령 어르신의 음성 발화 패턴과 복지 서비스 자체 FAQ를 구조화하여
Supabase에 저장하고, 기존 RAG 파이프라인에 통합한다.

**핵심 가치**: 현재 지식베이스(welfare_kb 29건)는 "서비스 중심"이고,
FAQ는 **"사용자 발화 중심"**이다.
→ "아파요"라는 짧은 음성 입력이 들어와도 FAQ 매칭을 통해
정확한 서비스 + 실용적 안내를 함께 제공할 수 있게 한다.

**완료 기준**:
- Supabase faq_kb 테이블 생성 + FAQ 데이터 N건 업로드
- RAG 파이프라인에서 welfare_kb + faq_kb 병렬 검색
- "아파요", "혼자 있어요" 같은 짧은 발화에도 관련 FAQ가 RAG 컨텍스트에 포함
- Gemini 응답에 FAQ 정보가 자연스럽게 녹아들어 안내

## 2. 배경

### 왜 필요한가
- 고령 어르신의 음성 입력은 짧고 모호함 ("아파요", "도와줘요")
- 현재 welfare_kb는 서비스 공식 정보(지원대상, 지원내용 등)로 구성
- 동일한 서비스라도 사용자가 궁금해하는 포인트는 다름:
  - welfare_kb: "긴급돌봄지원사업 - 지원대상: 65세 이상 독거노인..."
  - FAQ: "Q: 퇴원했는데 집에 아무도 없어요 → A: 긴급돌봄지원사업으로 24시간 내 돌봄 인력이 방문합니다. 055-230-8216으로 전화하세요."
- FAQ는 사용자 언어로 질문하고, 사용자 언어로 답변함

### 고령 어르신 발화 패턴 분류

| 패턴 유형 | 예시 발화 | 매핑 방향 |
|----------|----------|----------|
| **신체/건강 호소** | "아파요", "병원 가야 하는데", "약 받으러 가야 해" | 긴급돌봄, 종합재가, 의료연계 |
| **외로움/불안** | "아무도 없어요", "혼자 있는데", "무서워요" | 응급안전안심서비스, 노인맞춤돌봄 |
| **생활 곤란** | "밥을 못 해먹겠어", "집이 더러운데", "씻기가 힘들어" | 종합재가, 가사지원, AI 스마트돌봄 |
| **경제적 어려움** | "돈이 없어요", "병원비가 없어", "생활이 어려워" | 긴급복지지원, 통합돌봄 연계 |
| **절차 궁금** | "어디 전화해야 해요", "뭘 가져가야 해요", "신청 어떻게 해요" | 서비스별 신청절차 FAQ |
| **가족 대리** | "어머니가 혼자 사시는데", "아버지가 치매인 것 같아" | 노인맞춤돌봄, 치매안심센터 연계 |
| **서비스 이용 중** | "사람이 안 와요", "시간을 바꾸고 싶어요", "불만이 있어요" | 서비스 이용 관련 FAQ |
| **돌봄통합지원법** | "돌봄 서비스 통합된다던데", "새 법이 뭐예요" | v016 통합돌봄 KB 연계 |

## 3. 아키텍처 설계

### 3.1 데이터 구조

```
faq_kb 테이블 (Supabase)
├── id:              bigint (PK, auto)
├── category:        text    — 패턴 유형 (신체건강/외로움불안/생활곤란/경제어려움/절차궁금/가족대리/이용중/통합돌봄)
├── question:        text    — FAQ 질문 (사용자 발화 패턴)
├── answer:          text    — FAQ 답변 (실용적 안내)
├── related_service: text    — 연관 서비스명 (welfare_kb.service_name과 매칭)
├── keywords:        text    — 검색 키워드 (유의어, 방언, 구어체 포함)
├── phone_number:    text    — 바로 안내할 전화번호 (nullable)
├── priority:        int     — 표시 우선순위 (1=높음)
├── embedding:       vector(3072)  — gemini-embedding-001 임베딩
├── created_at:      timestamptz DEFAULT now()
└── updated_at:      timestamptz DEFAULT now()
```

### 3.2 RAG 통합 흐름

```
사용자 입력: "아파요"
     │
     ▼ embedContent(taskType: "RETRIEVAL_QUERY")
     │
     ├──→ match_welfare_kb() ──→ 서비스 결과 (긴급돌봄 0.78)
     │
     └──→ match_faq_kb()    ──→ FAQ 결과 ("아플 때 어떻게 하나요?" 0.85)
     │
     ▼ 병합 + 중복 제거
     │
     ▼ Gemini에 전달: welfare 컨텍스트 + FAQ 컨텍스트
     │
     ▼ AI 응답: 서비스 안내 + FAQ 실용 정보 통합
```


## 4. 작업 항목

### Phase 1: Supabase 테이블 + 함수 생성 (SQL Editor 수동 실행)

⚠️ anon key로 DDL 불가 → 아래 SQL을 출력하고 사용자에게 Supabase SQL Editor 실행 요청

#### SQL 1: faq_kb 테이블 생성

```sql
-- FAQ 지식베이스 테이블
CREATE TABLE IF NOT EXISTS faq_kb (
  id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  category text NOT NULL,
  question text NOT NULL,
  answer text NOT NULL,
  related_service text,
  keywords text,
  phone_number text,
  priority int DEFAULT 5,
  embedding vector(3072),
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);

-- 벡터 검색 인덱스
CREATE INDEX IF NOT EXISTS faq_kb_embedding_idx 
ON faq_kb 
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 10);

-- RLS 활성화
ALTER TABLE faq_kb ENABLE ROW LEVEL SECURITY;

-- 읽기 전용 정책 (anon key로 SELECT 가능)
CREATE POLICY "FAQ read access" ON faq_kb
  FOR SELECT USING (true);
```

#### SQL 2: match_faq_kb 함수 생성

```sql
CREATE OR REPLACE FUNCTION match_faq_kb(
  query_embedding vector(3072),
  match_threshold float DEFAULT 0.3,
  match_count int DEFAULT 5
)
RETURNS TABLE (
  id bigint,
  category text,
  question text,
  answer text,
  related_service text,
  keywords text,
  phone_number text,
  priority int,
  similarity float
)
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT
    f.id,
    f.category,
    f.question,
    f.answer,
    f.related_service,
    f.keywords,
    f.phone_number,
    f.priority,
    1 - (f.embedding <=> query_embedding) AS similarity
  FROM faq_kb f
  WHERE 1 - (f.embedding <=> query_embedding) > match_threshold
  ORDER BY f.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;
```

### Phase 3: server.js RAG 파이프라인 통합

#### 3-1. faq_kb 검색 함수 추가

파일: `server.js`

기존 `vectorSearchKnowledgeBase()` (또는 `pgvectorSearch()`) 근처에 FAQ 검색 함수 추가:

```javascript
// === FAQ 벡터 검색 ===
async function vectorSearchFAQ(queryText, topK = 3) {
  try {
    // 쿼리 임베딩 (기존 welfare_kb 검색과 동일한 방식)
    const embedding = await getQueryEmbedding(queryText);  // 기존 함수 재사용
    
    const { data, error } = await supabase.rpc('match_faq_kb', {
      query_embedding: embedding,
      match_threshold: 0.5,   // FAQ는 임계값 약간 높게 (노이즈 방지)
      match_count: topK,
    });
    
    if (error) {
      console.error('[FAQ 검색 오류]', error.message);
      return [];
    }
    
    return data || [];
  } catch (err) {
    console.error('[FAQ 검색 예외]', err.message);
    return [];
  }
}
```

⚠️ `getQueryEmbedding()`은 기존에 server.js에서 쿼리 임베딩을 생성하는 함수를 재사용.
   만약 별도 함수가 없으면 임베딩 코드를 공통 함수로 추출할 것.

#### 3-2. /api/chat 라우트에 FAQ 검색 통합

파일: `server.js`의 `/api/chat` 핸들러

기존 welfare_kb 검색 이후, FAQ 검색을 병렬로 추가:

```javascript
// 변경 전 (현재):
const welfareResults = await vectorSearchKnowledgeBase(cumulativeQuery);
// ... Gemini에 welfareResults만 전달

// 변경 후:
const [welfareResults, faqResults] = await Promise.all([
  vectorSearchKnowledgeBase(cumulativeQuery),

// FAQ 컨텍스트 구성

      (f.related_service ? `\n관련 서비스: ${f.related_service}` : '')

// Gemini 컨텍스트에 추가
const ragContext = welfareContext + faqContext;  // 기존 welfare 컨텍스트에 FAQ 추가
```

## 5. 주의사항

### 절대 변경 금지
- server.js 전체 구조 (단일 파일)
- withLock() 패턴
- keywordSearchKnowledgeBase() 폴백 함수
- Gemini 시스템 프롬프트 **기존 내용** (추가만 허용)
- 임베딩 모델명: gemini-embedding-001
- welfare_kb 테이블 및 match_welfare_kb 함수

### Supabase 제약
- Phase 1 SQL은 반드시 사용자에게 SQL Editor 실행 요청
- anon key로 faq_kb INSERT/SELECT은 가능 (RLS 정책 설정 후)
- INSERT 정책이 필요하면 Phase 1 SQL에 추가:
  ```sql
  CREATE POLICY "FAQ insert access" ON faq_kb
    FOR INSERT WITH CHECK (true);
  ```

### 성능 고려
- FAQ 검색은 welfare_kb 검색과 Promise.all로 병렬 실행 (지연 최소화)
- FAQ match_threshold를 0.5로 설정하여 노이즈 방지 (welfare_kb는 0.3)
- FAQ topK는 3건으로 제한 (Gemini 컨텍스트 과다 방지)

### 롤백 계획
- FAQ 기능은 welfare_kb와 독립적이므로 문제 시 FAQ 관련 코드만 제거/비활성화
- faqResults가 빈 배열이면 기존과 동일하게 동작 (graceful degradation)
