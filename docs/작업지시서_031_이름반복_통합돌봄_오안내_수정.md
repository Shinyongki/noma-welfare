## 문제 1: 이름 반복 질문

### 원인

1. chatHistory를 Gemini에 단일 문자열(`User: / Noma:` 접두어)로 직렬화하여 전달
   → Gemini가 역할 구분을 텍스트로만 판단해야 하므로 이전 대화 참조 불안정
2. `clean()` 함수에서 `<noma-apply>` 태그를 제거
   → Gemini 입장에서 신청이 완료됐는지, 이름이 수집됐는지 히스토리에서 확인 불가

### 수정 1-A. server.js — chatHistory를 Gemini contents 배열로 전달

**위치**: server.js:2151~2153 (Gemini 호출 시 userPrompt 구성 부분)

**현재 코드:**
```javascript
const userPrompt = ragContext + "\n\n" +
    sanitizedMessages.map(m => `${m.role === 'user' ? 'User' : 'Noma'}: ${m.content}`).join("\n");
```

현재 구조에서 Gemini `generateContentStream()` 호출 방식을 확인할 것.
만약 `contents` 파라미터에 단일 문자열을 넣고 있다면 아래와 같이 변경한다.

**변경 방향:**
- RAG 컨텍스트는 system instruction 또는 첫 user 메시지에 포함
- chatHistory는 `{role: 'user'|'model', parts: [{text}]}` 배열 형태로 전달
- 마지막 user 메시지에 RAG 컨텍스트를 prepend

실제 Gemini generateContentStream() 호출 코드를 먼저 출력하고, 현재 구조를 파악한 후
contents 배열 형태로 전환하는 방향으로 수정할 것.

**주의**: 이 수정은 영향 범위가 크므로 변경 전 현재 generateContentStream() 호출 코드 전체를
반드시 출력하여 보고한 후 수정할 것. 임의로 수정하지 말 것.

### 수정 1-B. code.html — noma-apply 수집 완료 정보를 히스토리에 보존

**위치**: code.html의 `clean()` 함수 (prepareMessagesForAPI 내부)

**현재 코드:**
```javascript
const clean = m => ({
    role: m.role,
    content: m.role === 'model'
        ? m.content.replace(/<noma-card>[\s\S]*?<\/noma-card>/g, '')
                   .replace(/<noma-apply>[\s\S]*?<\/noma-apply>/g, '').trim()
        : m.content
});
```

**변경 코드:**
```javascript
const clean = m => {
    if (m.role !== 'model') return { role: m.role, content: m.content };

    let content = m.content;
    // noma-card JSON은 제거 (길이가 길어 컨텍스트 낭비)
    content = content.replace(/<noma-card>[\s\S]*?<\/noma-card>/g, '');
    // noma-apply는 수집된 정보 요약으로 대체 (이름/전화번호 수집 완료 신호 보존)
    content = content.replace(/<noma-apply>([\s\S]*?)<\/noma-apply>/g, (_, json) => {
        try {
            const data = JSON.parse(json);
            return `[상담 신청 완료: 서비스="${data.serviceName}", 성함="${data.userName}", 전화="${data.userPhone}"]`;
        } catch {
            return '[상담 신청 완료]';
        }
    });
    return { role: m.role, content: content.trim() };
};
```

## 문제 2: 통합돌봄 신청 오안내

### 원인

1. 프롬프트 모순: 2078행("사회서비스원 상담 신청 안내하면 안 된다") vs 2086행("noma-apply로 접수 가능")
2. welfare_kb에 "통합돌봄 신청·연계 안내" 서비스가 등록되어 RAG가 소관 내로 매칭
   → Gemini가 신청 흐름(`<noma-apply>`) 진입

### 수정 2-A. server.js — v026 통합돌봄 프롬프트 모순 해소

**위치**: server.js:2069~2093 (v026 통합돌봄 시스템 프롬프트)

**현재 모순 구간 (2086행 부근):**
```
[통합돌봄 상담 신청 처리]
- 사용자가 통합돌봄 상담을 신청하려는 경우, 노마의 상담 신청 기능(noma-apply)을 통해 접수할 수 있다.
- 다만, 실제 통합돌봄 공식 신청은 읍면동 행정복지센터에서 해야 함을 반드시 안내한다.
```

**변경 내용:**
아래 내용으로 교체할 것.

```
[통합돌봄 상담 신청 처리 - 중요]
- 통합돌봄(돌봄통합지원법) 신청을 원하는 경우, 노마의 상담 신청(<noma-apply>) 기능을 사용하면 절대 안 된다.
- 반드시 읍면동 행정복지센터(주민센터) 또는 건강보험공단 지사에서 신청해야 함을 안내한다.
- 긴급돌봄이 필요한 경우에 한해 경상남도사회서비스원(055-230-8200) 직접 연락을 안내할 수 있다.
- 도민이 "신청하고 싶다"고 해도 noma-apply 신청 흐름으로 유도하지 않는다.
```

### 수정 2-B. server.js — noma-apply 생성 조건에 통합돌봄 제외 지시 추가

**위치**: server.js의 `<noma-apply>` 태그 생성 관련 시스템 프롬프트 지시 부분

현재 위치를 찾아 아래 지시를 추가할 것.

```
[noma-apply 생성 금지 조건]
아래 서비스 또는 상황에서는 <noma-apply> 태그를 절대 생성하지 않는다.
- 통합돌봄(돌봄통합지원법) 관련 신청
- 대분류가 "통합돌봄"인 welfare_kb 서비스
- 읍면동·건보공단 신청 안내를 제공해야 하는 모든 경우
```
