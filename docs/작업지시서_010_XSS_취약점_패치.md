# 작업지시서_010_XSS_취약점_패치

**목표:** admin.html에서 escapeHtml 없이 innerHTML에 삽입되는 사용자 데이터 8곳을 패치한다.

---

## 배경

공격 시나리오:
도민이 상담 신청 시 이름 필드에 `<img src=x onerror=alert(1)>` 입력
→ 관리자가 대시보드 또는 상세 모달을 열 때 XSS 실행

서버 측 sanitizeUserInput()은 프롬프트 인젝션만 필터링하고 HTML 태그는 필터링하지 않음.
escapeHtml() 함수는 이미 존재하며 (line 4215~4219) 82곳에서 사용 중.
8곳에서만 누락됨.

개선 범위: **stitch/admin.html 만 수정** (server.js 변경 없음)

---

## 작업 전 준비

```bash
cp stitch/admin.html stitch/admin.html.bak
```

### 위험도 높음 (7곳) — 도민 직접 입력값

### 위험도 중간 (1곳) — 담당자 입력값

## 작업 방법

grep으로 실제 줄 번호 재확인 후 패치:

```bash
# 패치 전 실제 위치 확인
grep -n "c\.userName\|data\.userName\|data\.userPhone\|data\.serviceName\|h\.actor\|h\.by" stitch/admin.html | grep -v "escapeHtml"
```

위 결과에서 `escapeHtml`이 없는 줄만 패치 대상.

---

## 검증

```bash
# 패치 후 잔여 취약점 확인 — 0개여야 정상
grep -n "innerHTML.*data\.userName\|innerHTML.*data\.userPhone\|innerHTML.*data\.serviceName" stitch/admin.html | grep -v "escapeHtml"

# 전체 escapeHtml 사용 횟수 확인 (82개 → 90개 예상)
grep -c "escapeHtml" stitch/admin.html
```

---

## 주의사항

- escapeHtml 함수 자체는 수정하지 말 것 (line 4215~4219)
- null/undefined 방어: `escapeHtml(value || '')` 패턴 사용
- 줄 번호는 작업 중 밀릴 수 있으므로 grep으로 재확인
- server.js 변경 없음
