# 작업지시서_022_복지무관_질의_가드레일
# 작성일: 2026-03-07

---

## 1. 목표

"날씨 어때?", "오늘 뭐 먹을까?", "축구 경기 결과 알려줘" 같은 **복지와 완전히 무관한 일상 질의**에 대해, 노마가 자연스럽게 자신의 역할을 안내하고 복지 검색으로 유도하도록 보강한다.

**핵심 구분**: 현재 v017의 "소관 외" 감지는 **복지이지만 우리 기관 소관이 아닌 질의**를 위한 것이고, 이번 작업은 **애초에 복지와 무관한 질의**를 처리하는 것이다. 이 두 가지는 명확히 다른 경로로 처리되어야 한다.

---

## 2. 배경

### 현재 상태
- 사용자가 "날씨 어때"를 입력하면:
  1. pgvector 서비스 검색 → 유사도 매우 낮음 (0.3~0.5 수준)
  2. pgvector FAQ 검색 → 역시 매우 낮음
  3. v017 소관 외 감지 → topScore < 0.72이므로 소관 외로 분류될 수 있음
  4. Gemini에 빈 RAG 컨텍스트 또는 소관 외 안내 지시가 전달됨
  5. 결과: **읍면동 전화번호를 안내하거나**, Gemini가 자체 판단으로 불확실한 응답

### 문제점
- 복지와 무관한 질의에 "읍면동 주민센터에 문의하세요" 같은 소관 외 안내가 뜨면 혼란
- Gemini가 날씨를 대답하려 시도할 수 있음 (Strict Grounding 위반)
- 사용자 경험 저하: "뭘 검색해야 하지?"라는 혼란 유발

### 해결 방향
1. **시스템 프롬프트에 비복지 질의 처리 가이드 추가** (Gemini 레벨)
2. **서버에서 사전 분류 로직 추가** (유사도가 극히 낮은 경우 별도 처리)

---

## 3. 작업 항목

### Phase A: 시스템 프롬프트 보강 (server.js)

**위치**: `defaultSystemPrompt` 끝부분 (v020 톤 가이드 뒤에 추가)

**추가할 내용** (기존 프롬프트 수정 없이 뒤에 이어붙이기):

```
═══ 비복지 질의 대응 가이드 ═══

[복지 무관 질의 감지]
사용자가 날씨, 뉴스, 스포츠, 음식 추천, 일반 상식, 계산, 번역 등 복지와 전혀 관련 없는 질문을 하면:

1. 부드럽게 역할을 밝힌다:
   - "저는 경상남도 복지 서비스 안내를 도와드리는 노마예요 😊"
   - 절대 해당 주제에 대한 답변을 시도하지 않는다

2. 자연스럽게 복지 검색으로 유도한다:
   - "혹시 돌봄이나 복지 서비스에 대해 궁금한 게 있으시면 편하게 물어봐 주세요!"
   - 예시를 하나 가볍게 제시: "예를 들어 '퇴원 후 돌봄이 필요해요'처럼 말씀해 주시면 돼요"

3. 톤은 현재 감지된 사용자 유형에 맞춘다:
   - 어르신 추정: "저는 복지 서비스 안내만 도와드릴 수 있어요~ 혹시 돌봄이나 건강 관련해서 궁금하신 거 있으세요?"
   - 일반 사용자: "저는 경상남도 복지 서비스 전문 안내 AI 노마입니다. 복지 관련 질문이 있으시면 도와드릴게요!"

4. 서비스 카드(<noma-card>)를 생성하지 않는다
5. 상담 신청(<noma-apply>)을 생성하지 않는다

[비복지 질의 예시]
- 날씨, 기온, 미세먼지
- 뉴스, 시사, 정치
- 스포츠 경기 결과
- 음식 추천, 맛집
- 번역, 영어 학습
- 수학 계산, 코딩
- 연예인, 드라마
- 길찾기, 교통 정보
- 쇼핑, 가격 비교

[주의: 복지 무관 vs 소관 외 구분]
- "날씨 어때?" → 복지 무관 → 위 가이드대로 역할 안내
- "기초연금 신청하려면?" → 소관 외 복지 → 읍면동/지자체 안내 (기존 소관 외 로직)
- "돌봄 서비스 알려줘" → 소관 내 복지 → 정상 RAG 응답
이 세 가지를 반드시 구분하여 응답한다.
```

### Phase B: 서버 사전 분류 로직 (server.js)

**위치**: `/api/chat` 라우트 내, pgvector 검색 결과 처리 부분

**로직**: topScore가 극히 낮은 경우 (예: 0.45 미만) Gemini 프롬프트에 비복지 힌트를 추가

```javascript
// 기존 코드에서 topScore 계산 후, 소관 외 분기 직전에 추가
const NON_WELFARE_THRESHOLD = 0.45;

// topScore가 극히 낮으면 → 복지와 아예 무관한 질의일 가능성 높음
let nonWelfareHint = '';
if (topScore < NON_WELFARE_THRESHOLD) {
  nonWelfareHint = `\n\n[시스템 참고: 이 질의는 복지 지식베이스와의 유사도가 매우 낮습니다(${topScore.toFixed(3)}). 복지와 무관한 일상 질의일 가능성이 높으니, "비복지 질의 대응 가이드"를 따라주세요. 소관 외 복지 안내(읍면동/주민센터)는 하지 마세요.]`;
}
```

이 `nonWelfareHint`를 Gemini에 보내는 시스템 메시지 또는 사용자 컨텍스트에 추가한다.

**주의**: 
- 기존 `OUT_OF_SCOPE_THRESHOLD = 0.72` 로직은 건드리지 않는다
- `NON_WELFARE_THRESHOLD = 0.45`는 0.72보다 훨씬 아래로, 두 임계값 사이(0.45~0.72)는 기존 소관 외 로직이 처리
- 분기 우선순위: `< 0.45` 비복지 > `< 0.72` 소관 외 > `>= 0.72` 소관 내

### Phase C: 쿼리 로깅 분류 추가 (server.js)

**위치**: v021 쿼리 로깅 부분

기존 `query_log` 테이블의 `match_category` 또는 유사 필드에 `non_welfare` 분류를 추가한다.

```javascript
// detectQueryCategory() 또는 기존 분류 로직에 추가
let queryCategory = 'in_scope'; // 기본값
if (topScore < NON_WELFARE_THRESHOLD) {
  queryCategory = 'non_welfare';
} else if (topScore < OUT_OF_SCOPE_THRESHOLD) {
  queryCategory = 'out_of_scope';
}
```

이를 통해 admin.html 분석 탭에서 "비복지 질의 비율"을 확인할 수 있다.

### Phase D: Filler 메시지 분기 (code.html, 선택사항)

비복지 질의는 RAG 검색 결과가 없으므로 Gemini 응답이 빠르게 올 수 있다. 기존 Filler 메시지 12종에 추가로, 매우 짧은 응답이 예상되는 경우의 처리가 자연스러운지 확인만 한다 (별도 변경 필요 없으면 스킵).

## 4. NON_WELFARE_THRESHOLD 값 근거

| 질의 유형 | 예상 topScore 범위 | 분류 |
|-----------|-------------------|------|
| "날씨 어때" | 0.25~0.40 | 비복지 |
| "오늘 뭐 먹을까" | 0.20~0.35 | 비복지 |
| "축구 경기 결과" | 0.15~0.30 | 비복지 |
| "기초연금 신청" | 0.55~0.70 | 소관 외 |
| "실업 급여" | 0.50~0.65 | 소관 외 |
| "돌봄 서비스" | 0.74~0.81 | 소관 내 |

임계값 0.45는 비복지와 소관 외 사이에 충분한 간격을 둔 값이다.

**단, 실제 배포 후 테스트 필수**: 
- "날씨" 등 5개 비복지 쿼리로 실제 topScore 확인
- 0.45가 너무 높거나 낮으면 조정

## 5. 검증 방법

### 비복지 질의 테스트 (5건)
| 입력 | 기대 결과 |
|------|----------|
| "날씨가 어때?" | 역할 안내 + 복지 검색 유도. 읍면동 안내 없음. 카드 없음 |
| "오늘 저녁 뭐 먹지" | 역할 안내 + 복지 검색 유도 |
| "BTS 신곡 나왔어?" | 역할 안내 + 복지 검색 유도 |
| "영어로 번역해줘" | 역할 안내 + 복지 검색 유도 |
| "1+1은 뭐야" | 역할 안내 + 복지 검색 유도 |

### 소관 외 질의 테스트 (기존 동작 유지 확인)
| 입력 | 기대 결과 |
|------|----------|
| "기초연금 신청하고 싶어요" | 소관 외 안내 (읍면동/주민센터) — 기존과 동일 |
| "실업급여 받을 수 있나요" | 소관 외 안내 — 기존과 동일 |

### 소관 내 질의 테스트 (기존 동작 유지 확인)
| 입력 | 기대 결과 |
|------|----------|
| "돌봄 서비스 알려주세요" | 정상 RAG 응답 + 서비스 카드 |
| "퇴원 후 집에서 돌봐줄 사람이 없어요" | 정상 RAG 응답 + 서비스 카드 |

### 인사 질의 테스트 (기존 동작 유지 확인)
| 입력 | 기대 결과 |
|------|----------|
| "안녕하세요" | 인사 응답 (기존과 동일) |
| "고마워요" | 감사 응답 (기존과 동일) |

### 쿼리 로그 확인
- Supabase query_log에 `non_welfare` 분류가 기록되는지 확인

## 6. 주의사항

1. **기존 코드 수정 금지**:
   - `defaultSystemPrompt` 기존 내용은 건드리지 않고, 끝에 추가만
   - `OUT_OF_SCOPE_THRESHOLD = 0.72` 변경 금지
   - v017 소관 외 분기 로직 변경 금지
   - v020 톤 가이드 변경 금지

2. **인사/일상 대화 기존 처리와 충돌 방지**:
   - 기존에 "안녕하세요" 등 인사 감지 로직이 있다면, 그것은 유지
   - "날씨"는 인사가 아니라 비복지 질의 → 별도 경로

3. **NON_WELFARE_THRESHOLD 값은 배포 후 조정 가능**:
   - 처음에 0.45로 시작, 실제 테스트 후 필요 시 0.40 또는 0.50으로 조정
   - 조정 시 이 문서의 표도 업데이트

4. **CLAUDE.md 업데이트**:
   - "비복지 질의 가드레일" 항목 추가 (NON_WELFARE_THRESHOLD = 0.45)
   - 3단계 분류 체계 명시: 비복지 < 0.45 / 소관 외 0.45~0.72 / 소관 내 >= 0.72
