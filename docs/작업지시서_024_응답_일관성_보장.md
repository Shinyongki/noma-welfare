# 작업지시서_024_응답_일관성_보장
# 작성일: 2026-03-07

---

## 1. 목표

같은 사용자가 같은(또는 유사한) 질의를 반복했을 때, 노마의 응답이 **핵심 정보(추천 서비스, 자격요건, 연락처) 수준에서 일관성을 유지**하도록 보장한다.

**일관성의 범위 정의**:
- ✅ 일관해야 하는 것: 추천 서비스 목록, 자격요건, 신청방법, 연락처, 혜택 내용
- ✅ 일관해야 하는 것: 서비스 A와 B의 차이점 설명
- 🔄 자연스럽게 달라져도 되는 것: 인사말, 공감 표현, 문장 구성, 부연 설명의 순서
- ❌ 달라지면 안 되는 것: 이전에 "가능하다"고 한 것을 "안 된다"고 바꾸는 것

## 2. 배경

### 현재 비일관성 발생 원인 분석

| 원인 | 영향도 | 제어 가능성 |
|------|--------|------------|
| Gemini temperature | 높음 — 같은 질문에 표현·구조·강조점 변동 | ✅ 높음 (파라미터 조정) |
| RAG 서비스 추천 순서 | 높음 — "첫 번째 추천"이 바뀌면 혼란 | ✅ 중간 (정렬 안정화) |
| FAQ 매칭 변동 | 중간 — 부가 정보 초점이 달라짐 | 🔶 중간 |
| 톤 분화 감지 | 낮음 — 말투가 좀 다른 정도 | 무시 가능 |
| 대화 맥락 누적 차이 | 중간 — 재방문 vs 이어서 질문 | 🔶 중간 |
| 지식베이스 업데이트 | 해당 없음 — 현재 정적 데이터 | — |

### 사용자가 느끼는 비일관성 시나리오

**시나리오 A: 추천 서비스 변동**
> 1차: "퇴원 후 돌봄" → 긴급돌봄지원사업, 종합재가센터, 통합돌봄센터
> 2차: "퇴원 후 돌봄" → 통합돌봄센터, 노인맞춤돌봄, 긴급돌봄지원사업
> 사용자: "아까랑 다른 서비스를 알려주네?"

**시나리오 B: 정보 충돌**
> 1차: "긴급돌봄 대상이 누구예요?" → "65세 이상 독거노인이 주요 대상이에요"
> 2차: "긴급돌봄 대상이 누구예요?" → "긴급한 돌봄이 필요한 모든 분이 대상이에요"
> 사용자: "아까는 65세 이상이라더니, 지금은 아무나 된다는 거야?"

**시나리오 C: 가능/불가능 판단 변동**
> 1차: "40살인데 돌봄 서비스 받을 수 있을까요?" → "가능성이 있어요"
> 2차: 같은 질문 → "주로 65세 이상 어르신을 위한 서비스예요"
> 사용자: "아까는 된다고 했잖아!"

## 3. 작업 항목

### Phase A: Gemini Temperature 최적화 (server.js)

**현재 상태 확인 필요**: server.js에서 Gemini 호출 시 temperature 파라미터가 명시되어 있는지 확인.

**목표값**: 
- 도민용 /api/chat: **temperature 0.3** (낮춰서 일관성 확보, 하지만 0은 너무 기계적)
- 담당자용 /api/staff/chat: **temperature 0.5** (분석·제안은 약간의 다양성이 유용)

**변경 위치**: server.js 내 Gemini API 호출 부분

```javascript
// 도민용 채팅 — 일관성 우선
const chatResponse = await model.generateContentStream({
  // 기존 파라미터 유지
  generationConfig: {
    temperature: 0.3,   // 핵심: 낮은 temperature로 일관성 확보
    // 기존 다른 설정 유지
  },
  // ...
});

// 담당자용 AI 상담 — 다양한 관점 허용
const staffResponse = await model.generateContentStream({
  generationConfig: {
    temperature: 0.5,
    // ...
  },
  // ...
});
```

**주의**: Gemini 2.0 Flash에서 temperature 파라미터가 generationConfig 내에 위치하는지, 최상위에 위치하는지 확인 필요. 현재 server.js 코드 구조에 맞게 적용할 것.

### Phase B: 세션 내 RAG 캐시 (server.js)

같은 세션에서 같은(또는 매우 유사한) 질의가 반복되면, 이전 RAG 검색 결과를 재사용한다.

#### B-1. 세션 캐시 구조

```javascript
// 세션별 RAG 캐시 (메모리, 서버 재시작 시 초기화)
const sessionRAGCache = new Map();

const RAG_CACHE_TTL = 10 * 60 * 1000;  // 10분
const RAG_CACHE_MAX_PER_SESSION = 20;    // 세션당 최대 20개 캐시

/**
 * 캐시 키 생성 — 질의를 정규화하여 유사 질의도 같은 키로 매핑
 */
function normalizeQueryForCache(query) {
  return query
    .replace(/[?？!！.。,，~\s]+/g, ' ')  // 문장부호/공백 정규화
    .trim()
    .toLowerCase();
}

/**
 * 세션 RAG 캐시에서 조회
 */
function getSessionRAGCache(sessionId, query) {
  const key = normalizeQueryForCache(query);
  const sessionCache = sessionRAGCache.get(sessionId);
  if (!sessionCache) return null;

  const cached = sessionCache.get(key);
  if (!cached) return null;

  // TTL 확인
  if (Date.now() - cached.timestamp > RAG_CACHE_TTL) {
    sessionCache.delete(key);
    return null;
  }

  return cached;
}

/**
 * 세션 RAG 캐시에 저장
 */
function setSessionRAGCache(sessionId, query, ragResults, faqResults) {
  const key = normalizeQueryForCache(query);

  if (!sessionRAGCache.has(sessionId)) {
    sessionRAGCache.set(sessionId, new Map());
  }

  const sessionCache = sessionRAGCache.get(sessionId);

  // 최대 개수 초과 시 가장 오래된 것 제거
  if (sessionCache.size >= RAG_CACHE_MAX_PER_SESSION) {
    const oldestKey = sessionCache.keys().next().value;
    sessionCache.delete(oldestKey);
  }

  sessionCache.set(key, {
    ragResults,
    faqResults,
    timestamp: Date.now(),
  });
}

/**
 * 세션 종료 시 캐시 정리 (주기적 실행)
 */
function cleanExpiredRAGCache() {
  const now = Date.now();
  for (const [sessionId, cache] of sessionRAGCache) {
    for (const [key, value] of cache) {
      if (now - value.timestamp > RAG_CACHE_TTL) {
        cache.delete(key);
      }
    }
    if (cache.size === 0) {
      sessionRAGCache.delete(sessionId);
    }
  }
}

// 5분마다 만료 캐시 정리
setInterval(cleanExpiredRAGCache, 5 * 60 * 1000);
```

#### B-2. /api/chat 라우트 내 캐시 적용

```javascript
// /api/chat 라우트 내, pgvector 검색 전에 캐시 확인
const sessionId = req.sessionID || req.ip || 'anonymous';
const cached = getSessionRAGCache(sessionId, userMessage);

let ragResults, faqResults;

if (cached) {
  // 캐시 히트 — 이전과 동일한 RAG 결과 사용
  ragResults = cached.ragResults;
  faqResults = cached.faqResults;
  console.log(`[RAG Cache] HIT for session ${sessionId}`);
} else {
  // 캐시 미스 — 정상 pgvector 검색
  [ragResults, faqResults] = await Promise.all([
    pgvectorSearch(queryEmbedding, ...),
    pgvectorFaqSearch(queryEmbedding, ...),
  ]);

  // 캐시 저장
  setSessionRAGCache(sessionId, userMessage, ragResults, faqResults);
  console.log(`[RAG Cache] MISS for session ${sessionId}`);
}

// 이후 기존 로직 동일하게 진행
```

**핵심**: 캐시는 **RAG 검색 결과(서비스 목록 + FAQ)**만 저장한다. Gemini 응답 자체는 캐시하지 않는다. 이렇게 하면:
- 추천 서비스 목록이 동일하게 유지됨 (시나리오 A 해결)
- Gemini에 동일한 컨텍스트가 주입되므로 정보 충돌 가능성 감소 (시나리오 B 완화)
- 문장 표현은 자연스럽게 약간 달라질 수 있음 (사용자에게 기계적으로 느껴지지 않음)

### Phase C: 시스템 프롬프트 일관성 지시 (server.js)

Gemini에게 **이전 대화 내용과의 일관성 유지**를 명시적으로 지시한다.

**위치**: `defaultSystemPrompt` 끝부분 (022, 023 가이드 뒤에 추가)

```
═══ 응답 일관성 유지 가이드 ═══

[핵심 원칙]
사용자와의 대화에서 이전에 안내한 정보와 모순되는 답변을 하지 않는다.

[서비스 정보 일관성]
- 같은 서비스에 대해 이전 대화에서 안내한 자격요건, 신청방법, 연락처를 변경하지 않는다.
- 이전에 추천한 서비스를 갑자기 빼거나, 새 서비스를 근거 없이 추가하지 않는다.
- 이전 대화에서 언급한 서비스 순서를 크게 바꾸지 않는다.

[가능성 표현 일관성]
- 이전에 "받을 수 있을 거예요"라고 안내한 것을 "해당되지 않아요"로 뒤집지 않는다.
- 추가 정보가 확인된 경우에만 판단을 보완한다: "말씀해 주신 상황을 더 고려하면..."
- 불확실한 경우: 이전 안내를 유지하면서 "정확한 확인은 담당 기관에서 해주실 수 있어요"로 보완한다.

[반복 질문 대응]
- 사용자가 같은 질문을 다시 하면, 이전 답변의 핵심 내용을 유지하되 표현을 약간 바꿔 반복한다.
- "아까 말씀드린 것처럼"으로 시작하여 자연스럽게 이전 안내를 재확인한다.
- 추가로 도움이 필요한 부분이 있는지 물어본다.

[모순 발견 시]
- RAG 컨텍스트에 이전 대화와 다른 정보가 포함된 경우, 이전 대화의 맥락을 우선한다.
- 명백한 오류를 발견한 경우에만 정정하되, "다시 확인해 보니"로 시작하여 정중하게 정정한다.
```

### Phase D: 이전 답변 컨텍스트 주입 (server.js)

현재 멀티턴 대화에서 이전 메시지는 Gemini에 전달되지만, **이전에 추천한 서비스 목록**이 명시적으로 전달되지 않을 수 있다. 이전 턴에서 추천한 서비스명을 별도로 추적하여 Gemini 컨텍스트에 주입한다.

#### D-1. 이전 추천 서비스 추적

```javascript
/**
 * 대화 이력에서 이전에 추천된 서비스명을 추출
 * chatHistory 배열에서 assistant 메시지의 <noma-card> 태그를 파싱
 */
function extractPreviousRecommendations(chatHistory) {
  const recommendations = [];

  for (const msg of chatHistory) {
    if (msg.role === 'assistant' || msg.role === 'model') {
      // <noma-card> 태그에서 서비스명 추출
      const cardMatches = msg.content?.matchAll(/<noma-card>([\s\S]*?)<\/noma-card>/g);
      if (cardMatches) {
        for (const match of cardMatches) {
          try {
            const card = JSON.parse(match[1]);
            if (card.serviceName) {
              recommendations.push(card.serviceName);
            }
          } catch (e) {
            // 파싱 실패 무시
          }
        }
      }
    }
  }

  // 중복 제거, 순서 유지
  return [...new Set(recommendations)];
}
```

#### D-2. Gemini 컨텍스트에 주입

```javascript
// /api/chat 라우트 내, Gemini 호출 직전
const previousRecs = extractPreviousRecommendations(chatHistory);

let consistencyHint = '';
if (previousRecs.length > 0) {
  consistencyHint = `\n\n[이전 대화에서 안내한 서비스: ${previousRecs.join(', ')}]\n이전에 안내한 서비스와의 일관성을 유지하세요. 같은 맥락의 질문이면 동일한 서비스를 우선 안내하세요.`;
}

// 기존 시스템 메시지 또는 사용자 컨텍스트에 consistencyHint 추가
```

### Phase E: RAG 결과 정렬 안정화 (server.js)

pgvector 유사도 점수가 동일하거나 매우 근접한 서비스들의 순서가 바뀌는 문제를 해결한다.

```javascript
/**
 * RAG 결과를 안정적으로 정렬
 * 유사도 점수가 같으면 서비스 ID로 2차 정렬하여 순서 고정
 */
function stableSortRAGResults(results) {
  return results.sort((a, b) => {
    // 유사도 내림차순 (소수점 3자리까지만 비교하여 미세한 차이 무시)
    const scoreDiff = Math.round(b.similarity * 1000) - Math.round(a.similarity * 1000);
    if (scoreDiff !== 0) return scoreDiff;

    // 유사도 동일 시 ID로 안정 정렬
    return (a.id || 0) - (b.id || 0);
  });
}
```

**적용 위치**: pgvectorSearch() 결과 반환 직전, 또는 /api/chat에서 결과를 받은 직후.

## 4. 변경 영향도 분석

| Phase | 변경 파일 | 위험도 | 기존 기능 영향 |
|-------|----------|--------|--------------|
| A (temperature) | server.js | 낮음 | 응답 다양성 약간 감소, 품질 영향 없음 |
| B (RAG 캐시) | server.js | 중간 | 메모리 사용 증가 (세션당 최대 20개, 10분 TTL) |
| C (프롬프트) | server.js | 낮음 | 기존 프롬프트 끝에 추가만 |
| D (이전 추천 추적) | server.js | 낮음 | chatHistory 파싱 추가, 기존 흐름 변경 없음 |
| E (정렬 안정화) | server.js | 낮음 | 기존 정렬 로직 대체 |

### 메모리 사용 추정

```
세션당 RAG 캐시: 최대 20건 × ~2KB = ~40KB
동시 활성 세션 100개 가정: ~4MB
→ 서버 메모리 영향 무시 가능
```

## 6. 주의사항

1. **Gemini 응답 자체는 캐시하지 않는다**: RAG 검색 결과만 캐시. Gemini 응답을 캐시하면 대화 맥락 반영이 안 되고, "기계적 반복"으로 느껴질 수 있음

2. **temperature 0으로 내리지 않는다**: 0이면 완전 결정적이지만, 공감 표현이 획일화되어 "기계랑 대화하는 느낌" 발생. 0.3이 일관성과 자연스러움의 균형점

3. **캐시는 세션 범위**: 다른 사용자의 캐시와 섞이지 않음. 세션 만료(5분 비활성) 시 캐시도 자연 소멸

4. **지식베이스 업데이트 시 캐시 무효화**: 향후 KB가 업데이트되면 전체 캐시를 초기화하는 로직 필요. 현재는 정적 데이터이므로 해당 없음

5. **기존 코드 수정 범위**:
   - temperature 추가/변경: Gemini 호출 부분만
   - RAG 캐시: /api/chat 앞부분에 캐시 로직 삽입
   - 프롬프트: 기존 끝에 추가만
   - 이전 추천 추적: /api/chat에서 Gemini 호출 직전 컨텍스트 추가
   - 안정 정렬: 기존 정렬 로직 대체 또는 후처리 추가

6. **normalizeQueryForCache의 한계**: "퇴원 후 돌봄"과 "퇴원하고 집에서 돌봄 서비스"는 다른 캐시 키가 됨. 완벽한 유사 질의 매칭은 현재 범위에서는 어려우며, 이 경우에는 temperature 0.3 + 안정 정렬 + 프롬프트 지시로 일관성을 유지

## 7. 작업 순서

1. Phase A: Gemini temperature 조정 (server.js, Gemini 호출 부분)
2. Phase E: RAG 결과 안정 정렬 (server.js, 검색 결과 후처리)
3. Phase C: 시스템 프롬프트 일관성 가이드 추가 (server.js, defaultSystemPrompt)
4. Phase D: 이전 추천 서비스 추적 + 컨텍스트 주입 (server.js, /api/chat)
5. Phase B: 세션 내 RAG 캐시 (server.js, /api/chat)
6. 테스트: 일관성 테스트 A~D + 성능 + 기존 기능
7. CLAUDE.md 업데이트

**순서 이유**: A→E→C는 독립적이고 위험이 낮아 먼저. D는 chatHistory 파싱이 필요해 중간. B(캐시)는 가장 복잡하므로 마지막.
