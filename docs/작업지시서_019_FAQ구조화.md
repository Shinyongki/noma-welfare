## 1. 목표

이용자 질문 유형별 FAQ를 구조화하고, pgvector에 FAQ 전용 임베딩을 추가하여
**"비용이 얼마예요?", "신청 자격이 되나요?"** 같은 반복 질문에 대해
RAG가 정확한 FAQ 답변을 우선 매칭하도록 개선한다.

### 기대 효과
- 통합돌봄 8개 서비스(현재 FAQ 0건) → FAQ 추가로 답변 품질 향상
- FAQ 질문 유형 태깅 → 유형별 분석·관리 가능
- FAQ 전용 임베딩 → 반복 질문 직접 매칭 (서비스 검색 없이 즉시 답변)
- RAG 컨텍스트 최적화 → 관련 FAQ만 선별 주입 (토큰 절약)

```javascript
// 전체 FAQ를 평면 텍스트로 RAG 컨텍스트에 주입
if (detail.faq?.length)
    ragContext += `   - FAQ: ${detail.faq.map(f => `Q.${f.q}→A.${f.a}`).join(' | ')}\n`;
```

**문제점:**
1. 모든 FAQ를 한 줄로 연결 → 토큰 낭비, 관련 없는 FAQ도 포함
2. FAQ 질문 유형 분류 없음 → 분석·관리 불가
3. FAQ 별도 임베딩 없음 → "비용 얼마예요?" 같은 질문이 서비스명 매칭에 의존
4. 통합돌봄 8개 서비스 FAQ 부재 → 2026.3.27 시행 대비 필요

### Phase A: 데이터 구조화 (welfare_kb_detail_v3.json 수정)

#### A-1. FAQ 질문 유형(category) 태그 추가

기존 68건 FAQ에 `category` 필드를 추가한다.

**질문 유형 분류 (7가지):**

| category | 설명 | 예시 질문 |
|----------|------|----------|
| `cost` | 비용·요금 | "비용이 얼마나 드나요?" |
| `eligibility` | 자격·대상 | "장기요양등급이 없어도 되나요?" |
| `howto` | 신청 방법·절차 | "어떻게 신청하나요?" |
| `duration` | 기간·횟수 | "얼마나 오랫동안 받을 수 있나요?" |
| `content` | 서비스 내용·범위 | "요양보호사가 와서 무엇을 해주나요?" |
| `proxy` | 대리·위임 | "가족이 대신 신청해도 되나요?" |
| `general` | 기타 일반 | "오늘 당장 신청하면 오늘 도움을 받을 수 있나요?" |

**변경 전:**
```json

  "a": "기초생활수급자, 차상위계층..."
}
```

**변경 후:**
```json
{
  "category": "cost",
  "q": "비용이 얼마나 드나요?",
  "a": "기초생활수급자, 차상위계층..."

```

분류 기준:
- 질문 텍스트에 "비용", "돈", "얼마", "무료", "요금" → `cost`
- "자격", "대상", "해당", "등급", "조건" → `eligibility`
- "신청", "방법", "어디서", "어떻게", "서류" → `howto`
- "기간", "얼마나", "오래", "몇 번", "횟수" → `duration`
- "무엇", "어떤", "내용", "서비스", "해주" → `content`

### Phase B: FAQ 전용 임베딩 (Supabase + upload 스크립트)

#### B-1. Supabase welfare_faq 테이블 생성

사용자에게 아래 SQL을 Supabase SQL Editor에서 실행 요청:

```sql
-- FAQ 전용 테이블
CREATE TABLE IF NOT EXISTS welfare_faq (
  id BIGSERIAL PRIMARY KEY,
  service_name TEXT NOT NULL,
  category TEXT NOT NULL DEFAULT 'general',
  question TEXT NOT NULL,
  answer TEXT NOT NULL,
  embedding vector(3072),
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 벡터 인덱스
CREATE INDEX IF NOT EXISTS welfare_faq_embedding_idx
  ON welfare_faq USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 10);

-- FAQ 검색 함수
CREATE OR REPLACE FUNCTION match_welfare_faq(
  query_embedding vector(3072),
  match_threshold float DEFAULT 0.5,
  match_count int DEFAULT 3
)
RETURNS TABLE (
  id bigint,
  service_name text,
  category text,
  question text,
  answer text,
  similarity float
)
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT
    wf.id,
    wf.service_name,
    wf.category,
    wf.question,
    wf.answer,
    1 - (wf.embedding <=> query_embedding) AS similarity
  FROM welfare_faq wf
  WHERE 1 - (wf.embedding <=> query_embedding) > match_threshold
  ORDER BY wf.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;
```

### Phase C: server.js RAG 파이프라인 개선

#### C-1. FAQ 벡터 검색 함수 추가

파일: `server.js`

```javascript
// pgvector FAQ 검색 (Supabase 연동 시)
async function pgvectorFaqSearch(query, topK = 3) {
    const { embeddings } = await ai.models.embedContent({
        model: 'gemini-embedding-001',
        contents: query,
        config: { taskType: 'RETRIEVAL_QUERY' },
    });
    const embedding = embeddings[0].values;
    const { data, error } = await supabase.rpc('match_welfare_faq', {
        query_embedding: embedding,
        match_count: topK,
        match_threshold: 0.65,
    });
    if (error) throw error;
    return data;
}
```

FAQ 유형 감지 (간단한 키워드 기반):
```javascript
function detectFaqCategory(userMessage) {
    const msg = userMessage.toLowerCase();
    if (/비용|돈|얼마|무료|요금|가격/.test(msg)) return 'cost';
    if (/자격|대상|해당|조건|등급|나이/.test(msg)) return 'eligibility';
    if (/신청|방법|어디서|어떻게|서류|절차/.test(msg)) return 'howto';
    if (/기간|얼마나|오래|몇\s?번|횟수|언제까지/.test(msg)) return 'duration';
    if (/무엇|어떤|내용|서비스|뭐|해주/.test(msg)) return 'content';
    if (/대신|대리|가족|위임|다른\s?사람/.test(msg)) return 'proxy';
    return null; // 감지 실패 → 전체 FAQ 포함
}
```

RAG 컨텍스트 주입 변경:
```javascript
// pgvector FAQ 직접 매칭 결과가 있으면 우선 주입
if (faqDirectMatch.length > 0) {
    ragContext += `\n[FAQ 직접 매칭 - 아래 답변을 우선 참고하세요]\n`;
    faqDirectMatch.forEach(f => {
        ragContext += `- [${f.service_name}] Q. ${f.question}\n  A. ${f.answer}\n`;
    });
}

// 서비스별 FAQ는 관련 유형만 선별
const detectedCategory = detectFaqCategory(lastUserMessage);
if (detail.faq?.length) {
    const relevantFaqs = detectedCategory
        ? detail.faq.filter(f => f.category === detectedCategory || f.category === 'general')
        : detail.faq.slice(0, 3); // 유형 미감지 시 상위 3건만
    if (relevantFaqs.length > 0) {
        ragContext += `   - FAQ: ${relevantFaqs.map(f => `Q.${f.q}→A.${f.a}`).join(' | ')}\n`;
    }
}
```

#### C-3. 임베딩 호출 최적화

FAQ 검색과 서비스 검색을 **동시 병렬 호출**하여 응답 지연 최소화:

```javascript
// 변경 전: 순차 호출
const pgResults = await pgvectorSearch(cumulativeQuery, 5);
// ... 이후 FAQ 검색

// 변경 후: 병렬 호출
const [pgResults, faqResults] = await Promise.all([
    pgvectorSearch(cumulativeQuery, 5),
    pgvectorFaqSearch(lastUserMessage, 3),
]);
```

단, 임베딩 생성도 병렬화:
```javascript
// 임베딩 2건을 동시 생성
const [serviceEmbedding, faqEmbedding] = await Promise.all([
    ai.models.embedContent({ model: 'gemini-embedding-001', contents: cumulativeQuery, config: { taskType: 'RETRIEVAL_QUERY' } }),
    ai.models.embedContent({ model: 'gemini-embedding-001', contents: lastUserMessage, config: { taskType: 'RETRIEVAL_QUERY' } }),
]);
```

> 이렇게 하면 임베딩 API 호출이 2건으로 늘지만 병렬이므로 체감 지연은 동일.

### Supabase 제약
- anon key로 DDL 불가 → Phase B-1 SQL은 사용자에게 실행 요청
- welfare_faq 테이블 생성 전에 upload 스크립트 실행하면 실패
- ivfflat 인덱스는 데이터 100건 미만에서도 동작하지만, 정확도를 위해 lists=10 유지

### Gemini API 제한
- FAQ 임베딩 업로드 시 92건 × 500ms = 약 46초 소요
- 런타임에 임베딩 2건 병렬 호출 → 분당 제한 주의
- 429 에러 시 3초 백오프 재시도 로직 유지

### 폴백 안전성
- Supabase 미연결 시 FAQ 벡터 검색 건너뛰기 (기존 키워드 폴백 그대로)
- pgvectorFaqSearch 실패 시 catch → 서비스 검색 결과의 FAQ만 사용
- welfare_kb_detail_v3.json에서 category 필드 없는 FAQ도 정상 처리 (하위 호환)
