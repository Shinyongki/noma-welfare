# 작업지시서_002_품질점검

**목표:** 현재 서비스의 주요 기능이 실제로 잘 동작하는지 코드 레벨에서 점검하고,
문제점·개선 필요 사항을 목록으로 정리해서 Claude AI에 보고한다.

---

## 배경

새 기능 개발 전에 현재 상태를 정확히 파악해야 한다.
코드를 직접 수정하지 말고, 읽고 분석하고 보고만 할 것.

---

## 작업 항목

### 1. RAG 엔진 점검 (server.js)

아래 항목을 코드에서 직접 확인하고 결과 보고:

**1-1. 유의어 사전 범위**
```bash
grep -n "synonymMap\|synonyms" server.js | head -30
```
- synonymMap에 등록된 키워드 총 개수
- 21개 서비스 중 synonymMap에 커버되지 않는 서비스가 있는지 확인
- 특히 아래 서비스들의 키워드 커버리지 확인:
  - 발달장애인긴급돌봄
  - 병원동행
  - 퇴원환자연계
  - 피해장애인쉼터

**1-2. 스코어링 가중치 확인**
```bash
grep -n "score\|weight\|가중\|*3\|*2\|*1" server.js | head -40
```
- 현재 가중치 구조 (사업명×N, 키워드×N, 대상×N, 내용×N) 정리
- 노이즈 필터 임계값 확인 (몇 % 이하 제외?)

**1-3. 멀티턴 대화 컨텍스트**
```bash
grep -n "history\|chatHistory\|messages\|8개\|slice" server.js | head -30
```
- 대화 이력 최대 유지 개수 확인 (8개?)
- 이전 대화 컨텍스트가 RAG 검색에 반영되는지 확인

---

### 2. 인증 및 보안 점검 (server.js)

**2-1. requireAuth 적용 누락 확인**
```bash
grep -n "app\.get\|app\.post\|app\.patch\|app\.delete" server.js | grep -v "requireAuth\|requireDeptAuth\|requireCaseAuth"
```
- 보호되어야 하는데 미들웨어 없는 라우트 있는지 확인
- /api/admin/*, /api/case/*, /api/dept/*, /api/staff/* 중 인증 없는 것 목록화

**2-2. 세션 설정 확인**
```bash
grep -n "session\|httpOnly\|sameSite\|secure\|maxAge" server.js | head -20
```
- httpOnly, sameSite, secure(production) 설정 모두 있는지 확인
- 세션 만료 시간 확인 (8시간?)

**2-3. 입력값 검증 확인**
```bash
grep -n "sanitize\|validate\|trim\|escape\|xss" server.js | head -20
```
- sanitizeUserInput() 실제 적용 범위 확인
- /api/chat, /api/service-request/connect에 적용되어 있는지

---

### 3. 이메일 발송 점검 (server.js)

**3-1. Resend API 연동 확인**
```bash
grep -n "resend\|RESEND\|sendEmail\|fetch.*resend" server.js | head -20
```
- sendEmail() 함수 에러 처리 확인 (실패 시 서버 크래시 여부)
- 이메일 발송 실패 시 상담 신청 자체가 실패하는지, 아니면 신청은 저장되는지

**3-2. DEFAULT_RECIPIENTS 확인**
```bash
grep -n "DEFAULT_RECIPIENTS\|recipients\|EMAIL" server.js | head -20
```
- 현재 기본 수신자 설정 확인
- 부서별 라우팅 실제 구현 여부 확인

---

### 4. 데이터 안정성 점검 (data/requestStore.mjs)

**4-1. 원자적 쓰기 확인**
```bash
grep -n "tmp\|rename\|writeFile\|backup\|\.bak" data/requestStore.mjs
```
- tmp→rename 방식 실제 구현 확인
- 백업(.bak) 생성 로직 확인
- 손상 복구 로직 확인

**4-2. 동시 쓰기 충돌 가능성**
```bash
grep -n "lock\|mutex\|queue\|concurrent" data/requestStore.mjs
```
- 동시 요청 시 파일 쓰기 충돌 방지 로직 있는지 확인

### 5. 프론트엔드 주요 기능 점검

**5-1. SSE 스트림 파싱 (stitch/code.html)**
```bash
grep -n "noma-card\|noma-apply\|EventSource\|SSE\|parseNoma" stitch/code.html | head -30
```
- `<noma-card>` 파싱이 청크 경계에서 안전한지 확인
- `<noma-apply>` 자동 신청 트리거 조건 확인
- SSE 연결 실패 시 에러 처리 확인

**5-2. TTS 폴백 (stitch/code.html)**
```bash
grep -n "SpeechSynthesis\|fallback\|tts\|edge-tts\|speechSynthesis" stitch/code.html | head -20
```
- Edge TTS 실패 → 브라우저 폴백 로직 실제 구현 확인

**5-3. 음성 입력 STT (stitch/code.html)**
```bash
grep -n "SpeechRecognition\|webkitSpeechRecognition\|lastInputWasVoice" stitch/code.html | head -20
```
- 음성 입력 후 자동 검색 타이머(300ms) 확인
- STT 실패 처리 확인

**5-4. 상태 스테퍼 (stitch/case.html)**
```bash
grep -n "status\|stepper\|open\|confirmed\|contacted\|connected\|closed" stitch/case.html | head -30
```
- 5단계 상태 전환 UI 정상 구현 확인
- 역행 방지 로직 프론트에도 있는지 확인

**5-5. 2단계 승인 UI (stitch/admin.html)**
```bash
grep -n "pending\|dept_approved\|approved\|rejected\|revision" stitch/admin.html | head -30
```
- 7가지 승인 상태 배지 모두 구현되어 있는지 확인
